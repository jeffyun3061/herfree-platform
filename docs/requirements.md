# Herfree Platform — 요구사항 정의서

## 1. 프로젝트 개요

Herfree Platform은 헤르페스 환우와 감염 불안을 겪는 사용자가 증상, 검사 경험, 치료 과정, 연애와 고지 고민, 감정적 불안을 안전하게 나눌 수 있도록 돕는 커뮤니티 중심 프라이빗 헬스케어 웹앱이다.

기존의 정보 공유 환경은 네이버 카페, 단톡방, 유튜브 댓글 등으로 분산되어 있으며, 운영 부재로 인한 다툼, 불안 조장, 허위 정보, 개인정보 노출, 정보 신뢰도 저하 문제가 발생할 수 있다. Herfree는 이러한 문제를 개선하기 위해 익명 기반 커뮤니티, 게시판 분리, 신고 및 숨김 관리, 전문가 정보 콘텐츠, 유튜버 콘텐츠, 제품 큐레이션을 하나의 서비스 안에서 제공한다.

## 2. 서비스 목표

### 2.1 사용자 목표

- 헤르페스 관련 정보를 구조화된 방식으로 확인할 수 있다.
- 검사 대기, 포비아, 확진 후 경험, 재발 관리, 연애 고민을 목적별 게시판에서 나눌 수 있다.
- 민감한 내용을 닉네임 또는 익명으로 작성할 수 있다.
- 다른 사용자에게 공감, 위로, 응원 반응을 받을 수 있다.
- 운영자와 전문가가 정리한 정보 콘텐츠를 확인할 수 있다.
- 면역 관리 및 생활 관리 제품을 큐레이션 형태로 확인할 수 있다.

### 2.2 운영자 목표

- 커뮤니티 게시글, 댓글, 신고 내역을 안정적으로 관리할 수 있다.
- 부적절한 글과 댓글을 삭제보다 숨김 처리 방식으로 운영할 수 있다.
- 유튜브 영상, 공지사항, 정보글, 제품 정보를 직접 등록하고 관리할 수 있다.
- 추후 전문가 계정, 제품 후기, 결제, PWA, 모바일 앱으로 확장 가능한 구조를 확보한다.

## 3. 핵심 서비스 방향

Herfree의 핵심은 제품 판매가 아니라 커뮤니티다. 제품은 사용자의 신뢰와 커뮤니티 체류가 쌓인 이후 자연스럽게 연결되는 큐레이션 영역으로 제공한다.

서비스 우선순위는 다음과 같다.

1. 커뮤니티
2. 증상 및 경험 공유
3. 신고 및 관리자 운영
4. 유튜버/전문가 정보 콘텐츠
5. 제품 큐레이션
6. 추후 웹앱/PWA/모바일 앱 확장

## 5. 사용자 권한

| 권한 | 설명 |
|------|------|
| `USER` | 일반 회원 |
| `CREATOR` | 유튜버 또는 운영자 콘텐츠 작성자 |
| `DOCTOR` | 인증 의사 또는 전문가 |
| `MODERATOR` | 커뮤니티 운영 관리자 |
| `ADMIN` | 전체 관리자 |

초기 MVP에서는 `USER`와 `ADMIN`을 중심으로 구현하되, 권한 enum과 DB 구조는 `CREATOR`, `DOCTOR`, `MODERATOR` 확장을 고려한다.

## 12. 데이터베이스 전략

### 12.1 엔진·환경

| 환경 | DB | 접속 |
|------|-----|------|
| 로컬 | MySQL 8 (Docker) | `localhost:3306`, DB `herfree_db`, 사용자 `herfree_user` |
| 운영 | AWS RDS MySQL 8 | VPC 내부, EC2 백엔드만 3306 인바운드 |

- **H2는 메인 DB로 사용하지 않는다.** 로컬·CI·운영 모두 MySQL 8.x를 기준으로 스키마·쿼리·마이그레이션을 맞춘다.
- 로컬 기동: `docker compose -f docker-compose.local.yml up -d` (`deployment.md` 참고)

### 12.2 스키마 관리 (ddl-auto)

| 환경 | 정책 | 비고 |
|------|------|------|
| 로컬 | `spring.jpa.hibernate.ddl-auto: update` | gitignored `application-local.yml`에서만 설정 |
| 운영 | `validate` 또는 `none` | **Flyway** migration으로 스키마 변경 |

운영에서 `ddl-auto=update` 또는 `create`는 사용하지 않는다.

### 12.3 비밀·설정 파일

- DB 비밀번호, JWT secret, RDS URL 등 **저장소에 커밋하지 않는다.**
- 로컬: `application-local.yml.example`을 복사해 `application-local.yml` 생성 (`.gitignore` 대상)
- 운영: EC2·GitHub Secrets·AWS Secrets Manager 등 **환경 변수**로 주입 (`deployment.md`)

## 13. 설계 원칙 (1–12)

1. **인증·프로필 분리:** `users`(인증·권한)와 `user_profiles`(닉네임·노출)를 분리한다.
2. **Soft delete 우선:** 게시글·댓글·콘텐츠·회원은 `status` 변경으로 숨김·삭제를 표현한다.
3. **익명 정책:** 게시글·댓글은 `is_anonymous`로 표시명만 익명 처리하고, 작성자 ID는 서버에 보관한다(관리·신고·권한 검증용).
4. **신고 중복 방지:** 동일 신고자·동일 대상(`target_type` + `target_id`) 조합은 DB unique 제약으로 1회만 허용한다.
5. **반응 중복 방지:** 동일 사용자·동일 대상·동일 `reaction_type` 조합은 unique 제약으로 1회만 허용한다.
6. **RESTful API:** URL에 동사를 넣지 않고 HTTP 메서드·리소스 경로로 표현한다 (`api-spec.md`).
7. **DTO 분리:** Entity를 API 응답으로 직접 노출하지 않는다 (`convention.md`).
8. **도메인 패키지:** `com.herfree` 하위에 `domain/auth`, `domain/user` 등 기능별 패키지 + `global` 횡단 관심사.
9. **계층형 아키텍처:** Controller → Service → Repository. 전면 DDD·이벤트 소싱은 1차에서 채택하지 않는다.
10. **MySQL 단일 엔진:** 개발·운영 DB 엔진을 MySQL 8로 통일한다(H2 메인 DB 금지).
11. **관리자 API 분리:** 운영·신고·숨김·콘텐츠 등록은 `/api/admin/**` prefix로 묶는다.
12. **확장 가능 enum:** `role`, `board_type`, `reaction_type`, `report.reason` 등은 문자열 enum으로 저장하고 MVP 이후 권한·게시판 확장을 고려한다.

## 14. 운영 정책

### 14.1 게시글 운영 정책

- 욕설, 비방, 혐오 표현이 포함된 게시글은 숨김 처리될 수 있다.
- 허위 의료 정보 또는 과도한 불안 조장 글은 관리자 검토 대상이 된다.
- 개인정보가 포함된 글은 관리자 판단에 따라 숨김 처리될 수 있다.
- 게시글과 댓글은 가능한 완전 삭제보다 상태값 변경 방식으로 관리한다.

### 14.2 신고 정책

- 회원은 게시글, 댓글을 신고할 수 있다.
- 동일 회원이 동일 대상을 중복 신고할 수 없다.
- 관리자는 신고를 승인 또는 반려할 수 있다.
- 신고 승인 시 대상 게시글 또는 댓글은 숨김 처리될 수 있다.

### 14.3 의료 정보 안내 정책

## 15. 기술 스택

### Backend

- Java 17 (권장·ADR-001 승인)
- Spring Boot 3.x
- Spring Security
- JWT
- Spring Data JPA
- MySQL
- Validation
- Lombok
- Swagger / SpringDoc

### Frontend

- Next.js
- TypeScript
- Tailwind CSS
- 모바일 웹앱 중심 UI
- 추후 PWA 확장 고려

### Infra

- GitHub Actions
- AWS EC2
- Docker
- Nginx
- AWS RDS MySQL
- AWS S3
- Vercel 또는 Nginx 기반 프론트 배포

## 16. 배포 방향

초기 운영은 다음 구조를 기준으로 한다.

- Frontend: Next.js, Vercel 배포 또는 추후 Nginx 배포
- Backend: Spring Boot, Docker 기반 AWS EC2 배포
- Database: AWS RDS MySQL 또는 초기 MySQL
- File Storage: AWS S3
- CI/CD: GitHub Actions
- Domain: 운영 도메인 연결
- HTTPS: SSL 적용

상세 환경·파이프라인은 `deployment.md`를 참고한다.

## 17. 성공 기준

1차 MVP의 성공 기준은 다음과 같다.

- 회원이 가입하고 로그인할 수 있다.
- 회원이 게시글과 댓글을 작성할 수 있다.
- 게시글/댓글에 공감 반응을 남길 수 있다.
- 부적절한 게시글/댓글을 신고할 수 있다.
- 관리자가 신고를 처리하고 게시글/댓글을 숨김 처리할 수 있다.
- 운영자가 정보글, 영상, 제품을 등록할 수 있다.
- 모바일 환경에서 앱처럼 사용할 수 있는 UI를 제공한다.
